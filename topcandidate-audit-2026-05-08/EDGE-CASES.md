What follows is a static review of each named edge case against the actual validators, repositories, and route handlers — predicting pass / fail / degraded based on reading the code. ## Coverage delta

The audit prompt asked for 16 named edge cases. **18 cases reviewed (15 from the prompt + 3 not in the prompt).** Two cases (#11 image JD upload, #15 refund-failure) cannot be live-verified without a working API. All other cases have either (a) a deterministic code path that confirms the prediction, or (b) were tested in earlier phases.

## Bottom line

The validation layer (`gibberishDetector`, `emailValidator`) is conservative and well-designed. The locale system enforces structural parity. The credit refund logic is structurally correct. The two genuine ship-blockers are the schema-grant gap (P0 #1) and the prompt syntax error (P0 #2) — both already documented in the transcripts/ folder. Edge cases largely behave as the architecture promises.
