# Spec 08 — Security

## Threat model

- **In-scope**: operator's phone is lost or borrowed; a curious bystander
  scrolls the History tab and sees customer phone numbers. Operator's APK is
  reverse-engineered.
- **Out-of-scope**: Android OS compromise, root, Mallory-on-the-network
  beyond TLS guarantees, social engineering the operator.

## Controls

### Secrets

- HMAC secret stored in **`flutter_secure_storage`** only. On Android this
  uses EncryptedSharedPreferences which is backed by Android Keystore.
- Never logged. The dispatcher's logging code redacts the `Authorization`
  and `X-Bkash-Webhook-Signature` headers.
- Never written to plain SharedPreferences, asset bundles, or environment
  variables in source.
- Settings UI displays "•••••••• (saved)" after first save; the secret is
  not read back to the UI.

### Webhook URL

- TLS required in release builds. Debug builds may accept `http://localhost`
  for testing.
- Validator regex: `^https?://.+/api/confirm-purchase$`.
- The URL itself is not secret but is still stored in secure storage so the
  whole config is in one place.

### Logging

- Use `developer.log` everywhere; no `print`.
- HTTP request bodies are never logged in full — the dispatcher logs only
  `trxId`, response status, and a sanitized error message. The full SMS body
  is kept in the local DB for audit but is **not** sent anywhere.
- No analytics SDK. No crash reporter. We have no way to leak data we don't
  collect.

### Lock screen

- v1 has an optional biometric/PIN gate on app open (see UI spec §3, item 6).
  When enabled, the History tab's customer phone numbers are protected from
  shoulder-surfers.
- The foreground service notification shows only "bKash watcher running",
  never customer phone numbers or amounts.

### Network

- HTTP client enforces TLS 1.2+ (default in modern Dart).
- Certificate pinning: not implemented. The webhook URL is operator-supplied
  and we have no way to bundle the right pin. If we ever add it, document
  the rotation procedure here.

### APK signing

- Debug APK is signed with the Flutter default debug key.
- Release APK uses a self-signed key generated by the operator. Keystore
  password is operator's responsibility; not stored in the repo.

## Non-controls (intentionally absent)

- No remote kill switch. The app is owned by the operator end-to-end.
- No anti-tampering checks. Reverse engineering the APK reveals only the
  hard-coded URL regex and the parser; no secrets are bundled.
- No rate limiting. The webhook can rate-limit on its own end if needed.
